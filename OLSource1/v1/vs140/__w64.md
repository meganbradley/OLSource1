# __w64
(Microsoft Specific) This keyword is obsolete. In versions of Visual Studio earlier than Visual Studio 2013, this lets you mark variables, so that when you compile with [/Wp64](../vs140/-wp64--detect-64-bit-portability-issues-.md) the compiler will report any warnings that would be reported if you were compiling with a 64-bit compiler.

 <CodeContentPlaceHolder>5\</CodeContentPlaceHolder>

## Remarks  
  
> [!IMPORTANT]
>  The [/Wp64](../vs140/-wp64--detect-64-bit-portability-issues-.md) compiler option and <CodeContentPlaceHolder>6\</CodeContentPlaceHolder> keyword are deprecated in Visual Studio 2010 and Visual Studio 2013 and removed starting in Visual Studio 2013. If you use the <CodeContentPlaceHolder>7\</CodeContentPlaceHolder> compiler option on the command line, the compiler issues [Command-Line Warning D9002](assetId:///c58b405b-0f26-434e-b57f-4f05e1ca81e6). The <CodeContentPlaceHolder>8\</CodeContentPlaceHolder> keyword is silently ignored. Instead of using this option and keyword to detect 64-bit portability issues, use a Visual C++ compiler that targets a 64-bit platform. For more information, see [64-Bit Programming with Visual C++](../vs140/configuring-programs-for-64-bit--visual-c---.md).  
  
 Any typedef that has <CodeContentPlaceHolder>9\</CodeContentPlaceHolder> on it must be 32 bits on x86 and 64 bits on x64.  
  
 To detect portability issues by using versions of the Visual C++ compiler earlier than Visual Studio 2010, the <CodeContentPlaceHolder>10\</CodeContentPlaceHolder> keyword should be specified on any typedefs that change size between 32 bit and 64 bit platforms. For any such type, <CodeContentPlaceHolder>11\</CodeContentPlaceHolder> must appear only on the 32-bit definition of the typedef.  
  
 The <CodeContentPlaceHolder>12\</CodeContentPlaceHolder> keyword is ignored if the compilation does not use <CodeContentPlaceHolder>13\</CodeContentPlaceHolder>.
