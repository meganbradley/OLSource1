# C6101
warning C6101: Returning uninitialized memory  
  
 A successful path through the function does not set the named <CodeContentPlaceHolder>0\</CodeContentPlaceHolder> parameter. This message is generated based on SAL annotations that indicate that the function in question always succeeds. A function that doesn’t return a success/failure indication should set all of its <CodeContentPlaceHolder>1\</CodeContentPlaceHolder> parameters because the analyzer assumes that the <CodeContentPlaceHolder>2\</CodeContentPlaceHolder> parameter is uninitialized data before the function is called, and that the function will set the parameter so that it’s no longer uninitialized. If the function does indicate success/failure, then the <CodeContentPlaceHolder>3\</CodeContentPlaceHolder> parameter doesn't have to be set in the case of failure, and you can detect and avoid the uninitialized location. In either case, the objective is to avoid the reading of an uninitialized location. If the function sometimes doesn’t touch an <CodeContentPlaceHolder>4\</CodeContentPlaceHolder> parameter that’s subsequently used, then the parameter should be initialized before the function call and be marked with the <CodeContentPlaceHolder>5\</CodeContentPlaceHolder> annotation, or the more explicit <CodeContentPlaceHolder>6\</CodeContentPlaceHolder> or <CodeContentPlaceHolder>7\</CodeContentPlaceHolder> when appropriate. "Partial success" can be handled with the <CodeContentPlaceHolder>8\</CodeContentPlaceHolder> annotation. For more information, see [Using SAL Annotations to Reduce C/C++ Code Defects](../vs140/using-sal-annotations-to-reduce-c-c---code-defects.md).
